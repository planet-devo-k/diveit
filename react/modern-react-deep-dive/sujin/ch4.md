# 1. 서버사이드 렌더링이란?

## 1. 싱글페이지 애플리케이션

### 싱글 페이지 애플리케이션(SPA; Single Page Application)

렌더링과 라우팅에 필요한 대부분의 기능을 서버가 아닌 브라우저의 자바스크립트에 의존하는 방식.

최초에 첫 페이지에서 데이터를 모두 러온 이후에는 페이지 전환을 위한 모든 작업이 자바스크립트와 브라우저의 history.pushState와 history.replaceState로 이뤄지기 때문에 페이지를 불러온 이후에는 서버에서 HTML을 내려받지 않고 하나의 페이지에서 모든 작업을 처리하므로 싱글 페이지 애플리케이션이라고 함

### 전통적인 방식의 애플리케이션과 싱글 페이지 애플리케이션의 작동 비교

- 전통적인 방식

페이지 전환이 발생할 때마다 새롭게 페이지를 요청하고, HTML 페이지를 다운로드해 파싱하는 작업을 거친다. 이 과정은 페이지를 처음부터 새로 그려야 해서 페이지 전환 시 부자연 스러운 모습을 보임.

- 싱글 페이지 애플리케이션

최초에 한번 모든 리소스를 다운로드하고 나면 이후 페이지를 전환할 때 추가로 리소스를 다운로드 하는 시간이 필요 없어짐. 경우에 따라 페이지 전체를 새로 렌더링 하는 것이 아닌 페이지 전환에 필요한 일부 영역만 다시 그리게 되므로 매끄러운 UI를 보여줄 수 있게 됨

### 싱글 페이지 렌더링 방식의 유행과 JAM 스택의 등장

과거 PHP나 JSP(JavaServer Pages)를 기반으로 대부분의 웹 애플리케이션이 만들어졌을 때는 대부분 서버사이드렌더링으로 이루어짐. 여기서 자바스크립트는 보조적인 수단으로 사용

자바스크립트가 다양한 작업을 수행하게 되면서 모듈화하는 방안이 논의됨. 그에 따라 CommonJS, AMD(Asynchronous Module Definition)이 등장. 이러한 자바스크립트 모듈화의 결실, 사용자 기기의 성능 향상과 인터넷 속도의 발전 등으로 자바스크립트에서 할 수 있는 일이 다양해짐

2010년경 자바스크립트 수준에서 MVx 프레임 워크를 구현하기 시작했고, 서버에서만 할수 있는 복잡한 작업을 할 수 있게 되면서 자바스크립트의 역할과 구조가 점점 커짐

Vue, React, Angular 의 시대가 오게되면서 싱글 페이지 렌더링이 인기를 얻게 됐다.

- JAM(JavaScript, API, Markup) 스택
  대부분의 작업이 자바스크립트에서 가능하기 때문에 프론트엔드는 자바스크립트와 마크업(HTML,CSS)를 미리 빌드해 두고 정적으로 사용자에게 제공하면 이후 작동은 모두 사용자의 클라이언트에서 실행되어 서버 확장성 문제에서 좀 더 자유로워질 수 있게 됨.
  JAM스택의 인기와 Node.js 고도화에 힘입어 API 서버 자체도 자바스크립트로 구현하는 구조가 인기를 끌기 시작함.

### 새로운 패러다임의 웹서비스를 향한 요구

많은 양의 리소스가 자바스크립트로 넘어오기 시작하던 때, 자바스크립트 코드의 규모도 점차 커짐. 이 시기에 있었던 폭발적인 기술발전으로 이런 문제고 해결될 것이라 기대하기도 했지만. 웹 애플리케이션에서 제공하는 자바스크립트 리소스의 크기와 수도 모두 증가하기 시작함. 눈에 띄는 점은 자바스크립트 파싱을 위해 CPU를 소비하는 시간이 증가했다는 것. ⇒ 자바스크립트에서 처리해야 하는 코드의 절대적인 양이 증가

## 2. 서버 사이드 렌더링이란?

최초에 사용자에게 보여줄 페이지를 서버에서 렌더링해 빠르게 사용자에게 화면을 제공하는 방식을 의미

웹페이지가 점점 느려지는 상황에 대한 문제의식을 싱글 페이지 애플리케이션의 태생적ㅇ니 한계에서 찾고, 이를 개선하고자 기존의 웹개발이 다시금 떠오르고 있음.

- SPA와 SSR의 차이
  웹페이지의 렌더링 책임을 어디에 두느냐.

### 서버사이드 렌더링의 장점

- 최초 페이지 진입이 비교적 빠르다
- 검색 엔진과 SNS 공유 등 메타데이터 제공이 쉽다
- 누적 레이아웃 이동이 적다
- 사용자의 디바이스 성능에 비교적 자유롭다
- 보안에 좀 더 안전하다

### 단점

- 소스코드를 작성할 때 항상 서버를 고려해야 한다
- 적절한 서버가 구축되어 있어야 한다
- 서비스 지연에 따른 문제

## 3. SPA와 SSR을 모두 알아야 하는 이유

### 서버 사이드 렌더링 역시 만능이 아님

클라이언트에서 발생하는 모든 무거운 작업을 서버에 미루고, 작어빙 모두 서버에서 이뤄진다고 해서 모든 성능 문제가 해결되는 것은 아님. 잘못된 웹페이지 설계는 성능을 해칠 뿐만 아니라 눈에 띄는 성능 개선도 얻지 못하고 관리포인트만 늘어나기만 하는 역효과를 낳을 수 있음.

### 싱글 페이지 애플리케이션과 서버사이드 렌더링

싱글 페이지 애플리케이션과 LAMP 스택과 같이 서버에서 모든 페이지를 각각 빌드하는 서버 사이드 렌더링 방식인 멀티 페이지 애플리케이션

- 가장 뛰어난 싱글 페이지 애플리케이션은 멀티 페이지 애플리케이션보다 낫다
- 평균적인 싱글 페이지 애플리케이션은 평균저긴 멀티 페이지 애플리케이션보다 느리다

### 현대의 서버 사이드 렌더링

프론트엔드 개발자는 서버에서의 렌더링, 그리고 클라이언트에서의 렌더링을 모두 이해해야 두가지 장점을 완벽하게 이해해야 두가지 장점을 완벽하게 취하는 제대로 된 웹서비스를 구축할 수 있음.

# 2. 서버 사이드 렌더링을 위한 리액트 API

## 1. renderToString

인수로 넘겨받은 리액트 컴포넌트를 렌더링해 HTML 문자열로 변환하는 함수. 서버 사이드 렌더링을 구현하는게 가장 기초적인 API. 최초의 페이지릉 HTML로 먼저 렌더링 하는 역할을 하는 함수. 클라이언트에서 실행되는 자바스크립트 코드를 포함시키거나 렌더링하는 역할까지 해주지는 않음. 필요한 자바스크립트 코드는 여기에서 생성된 HTML과는 별도로 제공해 브라우저에 제공되어야 함.

## 2. renderToStaticMarkup

renderToString 과 유사한 함수. 동일하게 리액트 컴포넌트를 기준으로 HTML 문자열을 만든다. 차이점은 루트요소에 추가한 data-reactroot와 같은 리액트에서만 사용하는 추가적인 DOM 속성을 ㅁ나들지 않는다는 점.

리액트에서만 사용하는 속성을 제거하면 결과물인 HTML 크기를 아주 약간이라도 줄일 수 있음.

리액트의 이벤트 리스너가 필요없는 완전히 순수한 HTML을 만들때만 사용됨.

## 3. renderToNodeStream

renderToString과 결과물이 완전히 동일하지만 두가지 차이점이 있음

1. renderToString과 renderToStaticMarkup은 브라우저에도 실행할 수 있지만 renderToNodeStream은 브라우저에서 사용하는 것이 불가능
2. renderToString의 결과물이 문자열인것에 반해 renderToNodeStream의 결과물은 Node.js의 ReadableStream

   ReadableStream: utf-8로 인코딩된 바이트 스트림. Node.js나 Deno, Bun 같은 서버 환경에서만 사용가능. string 을 얻기 위해서는 추가적인 처리가 필요

- renderToNodeStream이 필요한 이유
  - 스트림 : 큰 데이터를 다룰 때 데이터를 청크(chunk, 작은 단위)로 분할해 조금씩 가져오는 방식

## 4. renderToStaticNodeStream

renderToNodeStream과 제공하는 결과물을 똑같으나 renderToStaticMarkup과 마찬가지로 리액트 자바스크립트에 필요한 리액트 속성이 제공되지 않음. 마찬가지로 hydrate를 할 필요가 없는 순수 HTML 결과물이 필요할 때 사용하는 메서드

## 5. hydrate

renderToString 이나 renderToNodeStream으로 생성된 HTML 콘텐츠에 자바스크립트 핸들러나 이벤트를 붙이는 역할을 함.

- render와의 차이점
  이미 렌더링된 HTML이 있다는 가정하에 작업이 수행되고, 이 렌더링된 HTML을 기준으로 이벤트를 붙이는 작업만 실행.
