#### 프론트엔드 테스트 방법

- 유닛 테스트
- 통합테스트 (Integration Test)
- E2E(End to End Test)

#### TDD(Test Driven Development)가 무엇이며, 어떤 단계를 거쳐 진행되는지 설명해 주세요.

TDD는 코드를 작성하기 전에 테스트 코드를 먼저 작성하여, 테스트가 개발을 주도하게 하는 방법론입니다.

보통 Red-Green-Refactor라는 3단계 사이클을 반복합니다.

- Red: 실패하는 테스트 케이스를 먼저 작성합니다.
- Green: 테스트를 통과하기 위해 필요한 최소한의 코드를 작성합니다.
- Refactor: 테스트 통과를 유지하면서 코드의 중복을 제거하고 가독성을 높이는 등 구조를 개선합니다.

#### TDD를 적용했을 때 얻을 수 있는 가장 큰 이점은 무엇인가요?

크게 세 가지를 꼽을 수 있습니다.

- 설계 품질 향상
- 디버깅 비용 감소: 개발 단계에서 즉각적인 피드백을 받으므로 버그를 초기에 발견할 수 있습니다.
- 과감한 리팩토링 가능: 이미 작성된 테스트 코드가 안전망 역할을 해주기 때문에, 코드를 수정해도 기존 기능이 망가지지 않았음을 확신할 수 있습니다.

#### TDD가 항상 좋은 것은 아닐 텐데, 어떤 단점이 있을까요?

초기 개발 속도의 저하. 테스트 코드 유지보수 비용

#### 모든 코드에 대해 테스트를 작성해야 한다고 생각하시나요?

테스트는 '안정성'을 주지만 '비용'이 발생
비즈니스 로직이 복잡하거나 재사용성이 높은 공통 컴포넌트,
서비스의 핵심 기능에는 엄격한 테스트를 적용하되, 단순한 UI 레이아웃이나 변경이 잦은 프로토타입 단계에서는 테스트 작성보다 빠른 기능 구현에 집중하는 등 유연한 판단이 필요
