## 1. 개요 (Overview)

타입스크립트는 자바스크립트(JavaScript)를 더 안전하게 사용할 수 있도록 **'타입(Type)' 관련 기능을 추가한 언어**입니다. 완전히 새로운 언어가 아닌, 자바스크립트의 기능을 모두 포함하면서 그 위에 안전장치를 얹은 **확장판(Superset)**입니다.

## 2. 탄생 및 생태계 (Background)

- **출시:** 2012년 마이크로소프트(MS)의 '앤더스 하일스버그'(C# 창시자)에 의해 개발되었습니다.
- **성격:** 오픈소스 프로젝트로 운영되며 전 세계 개발자들의 기여로 빠르게 발전 중입니다.
- **위상:** - 2022년 설문 기준, 자바스크립트 개발자 100명 중 89명이 이미 사용 중인 '필수 언어'입니다.
  - 국내외 채용 시장에서 프론트엔드/백엔드 불문하고 필수 역량으로 자리 잡았습니다.

## 3. 핵심 특징: "자바스크립트의 확장판"

타입스크립트와 자바스크립트의 관계는 다음과 같이 정의할 수 있습니다.

- **문법적 호환:** 자바스크립트의 모든 문법을 그대로 사용할 수 있습니다.
- **타입 시스템 도입:** - **JS:** 변수 선언 시 타입을 지정하지 않음 (유연함).
  - **TS:** 변수명 뒤에 `: number`와 같이 타입을 명시하여 값의 종류를 고정함 (엄격함).
- **학습 곡선:** 이미 자바스크립트에 익숙하다면 '타입' 관련 문법만 추가로 배우면 되므로 진입 장벽이 낮습니다.

## 4. 왜 타입스크립트인가? (Necessity)

자바스크립트가 세상에서 가장 인기 있는 언어임에도 불구하고 타입스크립트가 등장한 이유는**"자바스크립트가 너무 잘 나가서”**입니다.

| 구분          | 자바스크립트 (JS)                             | 타입스크립트 (TS)                            |
| ------------- | --------------------------------------------- | -------------------------------------------- |
| **초기 목적** | 웹 브라우저 내 간단한 상호작용 처리           | 대규모 애플리케이션 개발 및 유지보수         |
| **설계 철학** | 유연함, 간결함, 빠른 작성 위주                | 안정성, 견고함, 버그 방지 위주               |
| **변화 계기** | **Node.js의 등장**으로 어디서든 실행 가능해짐 | 대규모 프로젝트에서 JS의 유연함이 독이 됨    |
| **핵심 가치** | 높은 생산성                                   | **안전장치(Type)**를 통한 대규모 협업 용이성 |

# 타입스크립트(TypeScript) 개론 및 타입 시스템

## 1. 개요 (Overview)

타입스크립트는 자바스크립트(JavaScript)에 **'타입(Type)' 안전장치**를 추가한 언어입니다. 자바스크립트의 모든 기능을 포함하는 **확장판(Superset)**으로서, 대규모 애플리케이션의 안정성을 확보하기 위해 탄생했습니다.

## 2. 타입 시스템의 이해 (Type System)

모든 프로그래밍 언어에는 값의 종류를 구분하고 검증하는 '타입 시스템'이 존재합니다. 타입스크립트를 이해하기 위해서는 기존 시스템과의 차이를 아는 것이 중요합니다.

### 2.1 동적 타입 시스템 (Dynamic Type System)

- **대상:** JavaScript, Python 등
- **특징:** 변수의 타입을 코드가 실행되는 도중(Runtime)에 결정합니다.
- **장점:** 매우 자유롭고 유연하며, 코드 작성이 간결합니다.
- **단점:** 실행 전에는 오류를 알 수 없어, 실제 서비스 중 프로그램이 갑자기 종료되는 치명적인 버그가 발생할 수 있습니다.

### 2.2 정적 타입 시스템 (Static Type System)

- **대상:** Java, C 등
- **특징:** 코드를 실행하기 전(Compile-time)에 모든 변수의 타입을 결정합니다.
- **장점:** 실행 전에 타입 오류를 발견할 수 있어 안정성이 높습니다.
- **단점:** 모든 변수에 일일이 타입을 정의해야 하므로 코드량이 늘어나고 번거롭습니다.

## 3. 타입스크립트의 해결책: "점진적 타이핑"

타입스크립트는 위 두 시스템의 장점을 결합한 **점진적 타입 시스템(Gradual Type System)**을 채택했습니다.

1. **실행 전 검사:** 정적 타입 시스템처럼 실행 전 타입 오류를 검사하여 안정성을 확보합니다.
2. **타입 추론(Type Inference):** 모든 변수에 타입을 직접 적지 않아도, 변수에 담긴 초기값을 기준으로 타입을 자동으로 알아냅니다. (동적 타입의 유연함 유지)
3. **결론:** "안전함"과 "편리함"을 동시에 잡은 독창적인 시스템입니다.

## 4. 왜 타입스크립트인가?

자바스크립트가 웹 브라우저를 넘어 **Node.js**를 통해 서버, 모바일, 데스크톱 앱까지 영역을 확장하면서 기존의 '너무 유연한' 특성이 오히려 독이 되었습니다.

| 구분          | 자바스크립트 (JS)            | 타입스크립트 (TS)          |
| ------------- | ---------------------------- | -------------------------- |
| **검사 시점** | 런타임 (실행 중)             | 컴파일 타임 (실행 전)      |
| **타입 결정** | 동적 (유동적)                | 점진적 (고정 + 추론)       |
| **안정성**    | 낮음 (예상치 못한 종료 위험) | 높음 (사전 오류 방지)      |
| **생산성**    | 초기 작성은 빠름             | 유지보수 및 협업 시 압도적 |

## 5. 타입스크립트의 동작 원리

일반적인 언어(Java 등)는 컴파일 시 **기계어(Bytecode)**로 변환되지만, 타입스크립트는 **자바스크립트**로 변환됩니다. 이 과정을 자세히 살펴보면 다음과 같습니다.

### 5.1 컴파일 과정 비교

- **일반 언어:** `Source Code` -> `AST(추상 문법 트리)` -> `Bytecode` -> 실행
- **타입스크립트:** `TS Code` -> `AST` -> **[타입 검사]** -> `JS Code` -> 실행

### 5.2 핵심 단계: 타입 검사 (Type Checking)

타입스크립트 컴파일러(TSC)의 가장 중요한 역할은 **AST 생성 후 수행하는 '타입 검사**'입니다.

1. **검사:** 코드를 AST로 변환한 뒤, 타입 오류가 있는지 확인합니다.
2. **실패 시:** 만약 타입 오류가 발견되면 **컴파일을 즉시 중단**하고 JS 파일을 생성하지 않습니다. (이것이 안전장치 역할)
3. **성공 시:** 오류가 없다면, AST를 바탕으로 자**바스크립트 코드로 변환(Transpile)**합니다. 이때 모든 **타입 정보는 제거**됩니다.

> Note: 결과물인 자바스크립트 파일에는 타입 관련 코드(interface, type alias 등)가 전혀 남아있지 않습니다. 따라서 브라우저나 Node.js는 아무 문제 없이 이를 실행할 수 있습니다.

**⚠️ 23년 10월 이후 수강하시는 분들 필독! (ts-node는 더 이상 작동하지 않습니다)**

23년 10월 Node.js의 LTS(장기 지원 버전)가 20대로 업데이트 되며 ts-node가 정상적으로 동작하지 않고 있습니다. 따라서 23년 10월 이후에 이 강의를 수강하시는 분들께서는 ts-node 대신 아래 링크에서 소개하는 **tsx**를 사용하시기 바랍니다.

## 6. Hello TS World

타입스크립트를 실제로 설치하고 실행하는 과정입니다.

### 6.1 초기 설정 (Setup)

새로운 프로젝트 폴더를 만들고 Node.js 패키지를 초기화합니다.

- `npm init` (기본값 설정)
- **필수 설치:** `npm i @types/node`
  - Node.js의 내장 기능(console 등)에 대한 타입 정보를 제공합니다. 미설치 시 타입 검사 오류가 발생할 수 있습니다.

### 6.2 컴파일러 설치 (Install Compiler)

타입스크립트 컴파일러(TSC)를 전역(Global)으로 설치하여 어디서든 사용할 수 있게 합니다.

- 명령어: `npm i -g typescript` (Mac은 `sudo` 필요할 수 있음)
- 확인: `tsc -v`

### 6.3 Node v20 이전(20버전에서는 동작 X)

가장 기본적인 방법으로, 변환 후 실행하는 2단계 과정을 거칩니다.

1. **작성:** `src/index.ts` 파일 생성
2. **컴파일:** `tsc src/index.ts` -> `src/index.js` 파일 생성됨
3. **실행:** `node src/index.js` -> 자바스크립트 파일 실행

### 6.4 Node v20 이후

개발 편의성을 위해 **한 번에 실행(변환+실행)**하는 도구를 사용합니다.

- **도구:** **TSX** (TypeScript Execute)
  - _참고: 과거 `ts-node`가 많이 쓰였으나, Node.js v20 이상에서 호환성 문제로 `tsx` 권장_
- **설치:** `npm i -g tsx`
- **실행:** `tsx src/index.ts` (JS 파일을 생성하지 않고 즉시 실행 결과 출력)

## 7. 컴파일러 옵션 설정 (Compiler Options)

타입스크립트 컴파일러(TSC)가 코드를 어떻게 변환하고 검사할지 세부 사항을 설정하는 방법입니다. 프로젝트 루트의 `tsconfig.json` 파일에서 관리합니다.

### 7.1 설정 파일 생성

- **자동 생성:** 터미널에 `tsc --init` 입력 시 기본 옵션이 주석 처리된 `tsconfig.json` 파일이 생성됩니다.
- **직접 설정:** 학습 시에는 빈 파일을 만들고 필요한 옵션만 직접 추가하는 방식을 권장합니다.

### 7.2 주요 옵션 상세

`compilerOptions` 객체 안에 설정하며, 주요 항목은 다음과 같습니다.

| 옵션명      | 설명 및 예시                       |
| ----------- | ---------------------------------- |
| **include** | 컴파일할 파일의 경로를 지정합니다. |

`"include": ["src"]` (src 폴더 내 모든 파일 컴파일) |
| **target** | 변환될 자바스크립트 코드의 버전을 설정합니다.

- `"ES5"`: 화살표 함수 등이 `function` 키워드로 변환됨 (구형 브라우저 호환)

- `"ESNext"`: 최신 문법 유지 |
  | **module** | 자바스크립트 모듈 시스템을 설정합니다.

- `"CommonJS"`: `require`, `exports` 문법 사용 (Node.js 기본)

- `"ESNext"`: `import`, `export` 문법 사용 |
  | **outDir** | 컴파일 결과물(`.js`)이 생성될 폴더를 지정합니다.

`"outDir": "dist"` 설정 시 소스(`src`)와 결과물(`dist`) 분리 가능 |
| **strict** | **[중요]** 타입 검사의 엄격함 수준을 설정합니다.

- `true`: 매개변수 타입 명시 강제 등 엄격한 모드 (권장)

- `false`: 느슨한 검사 (오류 일부 허용) |
  | **moduleDetection** | 모든 파일을 전역 모듈이 아닌 '로컬(독립) 모듈'로 취급하도록 강제합니다.

`"force"` 설정 시 `export` 키워드가 없어도 파일 간 변수명 충돌 방지 |

⚠ **[필독] 24년 이후 하나의 옵션을 더 추가해주셔야 합니다.**

@types 버전이 20버전 이상으로 업데이트되면서 특정 라이브러리에서 타입 검사 오류가 발생하고 있습니다.

## 1. 개요 (Overview)

타입스크립트는 자바스크립트(JavaScript)에 **'타입(Type)' 안전장치**를 추가한 언어입니다. 자바스크립트의 모든 기능을 포함하는 **확장판(Superset)**으로서, 대규모 애플리케이션의 안정성을 확보하기 위해 탄생했습니다.

## 2. 타입 시스템의 이해 (Type System)

모든 프로그래밍 언어에는 값의 종류를 구분하고 검증하는 '타입 시스템'이 존재합니다. 타입스크립트를 이해하기 위해서는 기존 시스템과의 차이를 아는 것이 중요합니다.

### 2.1 동적 타입 시스템 (Dynamic Type System)

- **대상:** JavaScript, Python 등
- **특징:** 변수의 타입을 코드가 실행되는 도중(Runtime)에 결정합니다.
- **장점:** 매우 자유롭고 유연하며, 코드 작성이 간결합니다.
- **단점:** 실행 전에는 오류를 알 수 없어, 실제 서비스 중 프로그램이 갑자기 종료되는 치명적인 버그가 발생할 수 있습니다.

### 2.2 정적 타입 시스템 (Static Type System)

- **대상:** Java, C 등
- **특징:** 코드를 실행하기 전(Compile-time)에 모든 변수의 타입을 결정합니다.
- **장점:** 실행 전에 타입 오류를 발견할 수 있어 안정성이 높습니다.
- **단점:** 모든 변수에 일일이 타입을 정의해야 하므로 코드량이 늘어나고 번거롭습니다.

## 3. 타입스크립트의 해결책: "점진적 타이핑"

타입스크립트는 위 두 시스템의 장점을 결합한 **점진적 타입 시스템(Gradual Type System)**을 채택했습니다.

1. **실행 전 검사:** 정적 타입 시스템처럼 실행 전 타입 오류를 검사하여 안정성을 확보합니다.
2. **타입 추론(Type Inference):** 모든 변수에 타입을 직접 적지 않아도, 변수에 담긴 초기값을 기준으로 타입을 자동으로 알아냅니다. (동적 타입의 유연함 유지)
3. **결론:** "안전함"과 "편리함"을 동시에 잡은 독창적인 시스템입니다.

## 4. 왜 타입스크립트인가?

자바스크립트가 웹 브라우저를 넘어 **Node.js**를 통해 서버, 모바일, 데스크톱 앱까지 영역을 확장하면서 기존의 '너무 유연한' 특성이 오히려 독이 되었습니다.

| 구분          | 자바스크립트 (JS)            | 타입스크립트 (TS)          |
| ------------- | ---------------------------- | -------------------------- |
| **검사 시점** | 런타임 (실행 중)             | 컴파일 타임 (실행 전)      |
| **타입 결정** | 동적 (유동적)                | 점진적 (고정 + 추론)       |
| **안정성**    | 낮음 (예상치 못한 종료 위험) | 높음 (사전 오류 방지)      |
| **생산성**    | 초기 작성은 빠름             | 유지보수 및 협업 시 압도적 |

## 5. 타입스크립트의 동작 원리

일반적인 언어(Java 등)는 컴파일 시 **기계어(Bytecode)**로 변환되지만, 타입스크립트는 **자바스크립트**로 변환됩니다. 이 과정을 자세히 살펴보면 다음과 같습니다.

### 5.1 컴파일 과정 비교

- **일반 언어:** `Source Code` -> `AST(추상 문법 트리)` -> `Bytecode` -> 실행
- **타입스크립트:** `TS Code` -> `AST` -> **[타입 검사]** -> `JS Code` -> 실행

### 5.2 핵심 단계: 타입 검사 (Type Checking)

타입스크립트 컴파일러(TSC)의 가장 중요한 역할은 **AST 생성 후 수행하는 '타입 검사**'입니다.

1. **검사:** 코드를 AST로 변환한 뒤, 타입 오류가 있는지 확인합니다.
2. **실패 시:** 만약 타입 오류가 발견되면 **컴파일을 즉시 중단**하고 JS 파일을 생성하지 않습니다. (이것이 안전장치 역할)
3. **성공 시:** 오류가 없다면, AST를 바탕으로 자**바스크립트 코드로 변환(Transpile)**합니다. 이때 모든 **타입 정보는 제거**됩니다.

> Note: 결과물인 자바스크립트 파일에는 타입 관련 코드(interface, type alias 등)가 전혀 남아있지 않습니다. 따라서 브라우저나 Node.js는 아무 문제 없이 이를 실행할 수 있습니다.

**⚠️ 23년 10월 이후 수강하시는 분들 필독! (ts-node는 더 이상 작동하지 않습니다)**

23년 10월 Node.js의 LTS(장기 지원 버전)가 20대로 업데이트 되며 ts-node가 정상적으로 동작하지 않고 있습니다. 따라서 23년 10월 이후에 이 강의를 수강하시는 분들께서는 ts-node 대신 아래 링크에서 소개하는 **tsx**를 사용하시기 바랍니다.

## 6. Hello TS World

타입스크립트를 실제로 설치하고 실행하는 과정입니다.

### 6.1 초기 설정 (Setup)

새로운 프로젝트 폴더를 만들고 Node.js 패키지를 초기화합니다.

- `npm init` (기본값 설정)
- **필수 설치:** `npm i @types/node`
  - Node.js의 내장 기능(console 등)에 대한 타입 정보를 제공합니다. 미설치 시 타입 검사 오류가 발생할 수 있습니다.

### 6.2 컴파일러 설치 (Install Compiler)

타입스크립트 컴파일러(TSC)를 전역(Global)으로 설치하여 어디서든 사용할 수 있게 합니다.

- 명령어: `npm i -g typescript` (Mac은 `sudo` 필요할 수 있음)
- 확인: `tsc -v`

### 6.3 Node v20 이전(20버전에서는 동작 X)

가장 기본적인 방법으로, 변환 후 실행하는 2단계 과정을 거칩니다.

1. **작성:** `src/index.ts` 파일 생성
2. **컴파일:** `tsc src/index.ts` -> `src/index.js` 파일 생성됨
3. **실행:** `node src/index.js` -> 자바스크립트 파일 실행

### 6.4 Node v20 이후

개발 편의성을 위해 **한 번에 실행(변환+실행)**하는 도구를 사용합니다.

- **도구:** **TSX** (TypeScript Execute)
  - _참고: 과거 `ts-node`가 많이 쓰였으나, Node.js v20 이상에서 호환성 문제로 `tsx` 권장_
- **설치:** `npm i -g tsx`
- **실행:** `tsx src/index.ts` (JS 파일을 생성하지 않고 즉시 실행 결과 출력)

## 7. 컴파일러 옵션 설정 (Compiler Options)

타입스크립트 컴파일러(TSC)가 코드를 어떻게 변환하고 검사할지 세부 사항을 설정하는 방법입니다. 프로젝트 루트의 `tsconfig.json` 파일에서 관리합니다.

### 7.1 설정 파일 생성

- **자동 생성:** 터미널에 `tsc --init` 입력 시 기본 옵션이 주석 처리된 `tsconfig.json` 파일이 생성됩니다.
- **직접 설정:** 학습 시에는 빈 파일을 만들고 필요한 옵션만 직접 추가하는 방식을 권장합니다.

### 7.2 주요 옵션 상세

`compilerOptions` 객체 안에 설정하며, 주요 항목은 다음과 같습니다.

| 옵션명      | 설명 및 예시                       |
| ----------- | ---------------------------------- |
| **include** | 컴파일할 파일의 경로를 지정합니다. |

`"include": ["src"]` (src 폴더 내 모든 파일 컴파일) |
| **target** | 변환될 자바스크립트 코드의 버전을 설정합니다.

- `"ES5"`: 화살표 함수 등이 `function` 키워드로 변환됨 (구형 브라우저 호환)

- `"ESNext"`: 최신 문법 유지 |
  | **module** | 자바스크립트 모듈 시스템을 설정합니다.

- `"CommonJS"`: `require`, `exports` 문법 사용 (Node.js 기본)

- `"ESNext"`: `import`, `export` 문법 사용 |
  | **outDir** | 컴파일 결과물(`.js`)이 생성될 폴더를 지정합니다.

`"outDir": "dist"` 설정 시 소스(`src`)와 결과물(`dist`) 분리 가능 |
| **strict** | **[중요]** 타입 검사의 엄격함 수준을 설정합니다.

- `true`: 매개변수 타입 명시 강제 등 엄격한 모드 (권장)

- `false`: 느슨한 검사 (오류 일부 허용) |
  | **moduleDetection** | 모든 파일을 전역 모듈이 아닌 '로컬(독립) 모듈'로 취급하도록 강제합니다.

`"force"` 설정 시 `export` 키워드가 없어도 파일 간 변수명 충돌 방지 |

따라서 tsconfig.json에 compilerOption 내부에 **skipLibCheck** 옵션을 추가 해 주시기 바랍니다
